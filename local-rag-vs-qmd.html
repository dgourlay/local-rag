local-rag vs. qmd

Two on-device hybrid retrieval systems that expose search to LLM agents via MCP. Same niche, materially different design philosophies. This page tries to be honest: not every "yes" is a win, and "not documented" is noted as such rather than treated as "no."

github.com/dgourlay/local-rag github.com/tobi/qmd
How to read this page. More features is not the same as better. local-rag's pipeline is deliberately elaborate; qmd's is deliberately minimal. Both are legitimate design choices. Where a capability adds real cost (extra dependencies, latency, index-time LLM calls), we say so. Where a feature couldn't be confirmed from public sources, it is marked Not confirmed rather than No.

local-rag

Python 3.11+ · Qdrant (Docker) · BGE-M3 · Apache-2.0 · 0 stars (new)

Broader format coverage and a deeper retrieval pipeline. Strong fit if your corpus is mixed PDF/DOCX and you already live in the Claude/Kiro ecosystem.

  • StrengthPDF, DOCX, OCR via Docling
  • StrengthPyramid summaries, HyDE, recency boost, citation expansion
  • CostRequires Docker and an external LLM CLI (claude/kiro-cli)
  • CostNo published package; clone + make setup to install
  • CostMore complex pipeline has known latency suspects still under investigation

qmd (by Tobi Lütke)

TypeScript · Node ≥22 / Bun · SQLite + sqlite-vec · ~22k stars · v2.1.0

Smaller surface, self-contained runtime, strong developer ergonomics. Designed around markdown and source code, with distribution and polish well ahead of local-rag.

  • StrengthNo Docker, no external LLM CLI — node-llama-cpp runs everything in-process
  • StrengthPublished on npm, SDK mode (QMDStore), Raycast + OpenClaw integrations
  • StrengthAST-aware code chunking (tree-sitter), fine-tuned 1.7B query-expansion model
  • CostNo PDF, DOCX, or OCR ingestion
  • CostSimpler ranking (fewer lanes, no recency signal, no summary hierarchy)

Honest Tradeoff Notes

Complexity is a cost, not a feature. local-rag runs more stages: auto-question generation, pyramid summarization, HyDE, a 4-lane fusion, reranker enrichment, and citation expansion. Each adds either index-time or query-time latency, and several require an external LLM CLI. That pipeline may improve retrieval on some corpora; it may also be overkill for simple notes.
qmd's minimalism is deliberate. Fewer moving parts means simpler installs, fewer failure modes, and a self-contained runtime. Its single retrieval lane on chunks, expanded by a purpose-trained query-expansion model, is a coherent design — not an omission.
Things local-rag lacks that are not easy to copy. A fine-tuned query expansion model, AST-aware chunking via tree-sitter, an embeddable SDK, a published npm package, and existing ecosystem integrations (Raycast, OpenClaw). These are distribution and product-surface gaps, not algorithm gaps.
Things qmd lacks that are not easy to copy. Office-document ingestion (PDF, DOCX) and OCR — Docling is a substantial dependency, and getting it right in-process on Node/Bun is non-trivial. If your corpus is primarily office documents, this is a decisive difference.
Neither project publishes retrieval benchmarks. Every "better retrieval" claim on either side is unverified. A shared eval set with precision/recall numbers would settle more of this comparison than any feature list.
